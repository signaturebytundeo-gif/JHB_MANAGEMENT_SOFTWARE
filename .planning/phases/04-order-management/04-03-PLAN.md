  <files>apps/command-center/src/app/actions/operator-orders.ts</files>
  <action>
Add these server actions to the existing `operator-orders.ts` file:

1. `confirmOrder(orderId: string)`:
   - Call `verifyManagerOrAbove()` from dal.ts
   - Wrap everything in `db.$transaction(async (tx) => { ... })`
   - Fetch order with lineItems inside the transaction
   - Guard: order must be in DRAFT status, throw if not
   - For each line item, call `allocateInventoryFIFO(line.productId, order.locationId, line.quantity, tx as typeof db)` — this throws if insufficient stock
   - For each allocation result, create `tx.inventoryMovement.create()` with: movementType 'DEDUCTION', batchId from allocation, fromLocationId = order.locationId, toLocationId = null, quantity from allocation, orderId = order.id, requiresApproval = false, approvedById = session.userId, approvedAt = now, createdById = session.userId
   - Query `tx.approvalThreshold.findMany()` to get thresholds from DB (do NOT hardcode $150/$500)
   - Determine approvalStatus based on totalAmount vs thresholds: if below lowest threshold -> 'auto' (set status to CONFIRMED), if between -> 'pending_single', if above highest -> 'pending_dual'
   - Update order: set approvalStatus, if auto-approved set status=CONFIRMED + approvedById + approvedAt, otherwise keep DRAFT
   - `revalidatePath('/dashboard/orders')`
   - Return { success, message }

2. `approveOrder(orderId: string)`:
   - Call `verifyManagerOrAbove()`
   - Fetch order
   - If approvalStatus is 'pending_single': set approvedById, approvedAt, approvalStatus='approved', status='CONFIRMED'
   - If approvalStatus is 'pending_dual' and no first approver: set approvedById, approvalStatus stays 'pending_dual' (still needs second)
   - If approvalStatus is 'pending_dual' and first approver exists: ensure current user !== first approver (dual control), set secondApprovedById, approvalStatus='approved', status='CONFIRMED'
   - revalidatePath, return result

3. `updateOrderStatus(orderId: string, newStatus: string)`:
   - Call `verifyManagerOrAbove()`
   - Fetch order, validate transition is legal:
     - CONFIRMED -> PROCESSING
     - PROCESSING -> SHIPPED
     - SHIPPED -> DELIVERED
     - DELIVERED -> COMPLETED
     - Any status -> CANCELLED (except COMPLETED)
   - Reject invalid transitions with error message
   - Update order status
   - revalidatePath, return result

4. `getPickPackList(orderId: string)`:
   - Call `verifySession()`
   - Query `db.inventoryMovement.findMany()` where orderId = orderId AND movementType = 'DEDUCTION'
   - Include: batch (select batchCode, product select name/sku), fromLocation (select name)
   - Group by product: for each product, list the batch codes and quantities to pick
   - Return structured data: { orderId, orderNumber, location, items: [{ productName, sku, totalQuantity, batches: [{ batchCode, quantity }] }] }
   - This shows warehouse staff exactly which batches to pull (FIFO integrity)
