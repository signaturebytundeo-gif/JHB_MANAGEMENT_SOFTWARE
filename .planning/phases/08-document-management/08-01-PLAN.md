2. **schema.prisma** — Add DocumentCategory enum and two new models:

```
enum DocumentCategory {
  AGREEMENT
  INVOICE
  CERTIFICATION
  SOP
  MARKETING
  TEMPLATE
  OTHER
}

model Document {
  id               String            @id @default(cuid())
  name             String
  category         DocumentCategory
  currentBlobUrl   String
  isTemplate       Boolean           @default(false)

  // Polymorphic nullable FKs — at most one is non-null
  customerId       String?
  orderId          String?
  batchId          String?

  uploadedById     String
  createdAt        DateTime          @default(now())
  updatedAt        DateTime          @updatedAt

  customer         Customer?         @relation(fields: [customerId], references: [id])
  order            Order?            @relation(fields: [orderId], references: [id])
  batch            Batch?            @relation(fields: [batchId], references: [id])
  uploadedBy       User              @relation("DocumentUploader", fields: [uploadedById], references: [id])
  versions         DocumentVersion[]
}

model DocumentVersion {
  id             String   @id @default(cuid())
  documentId     String
  versionNumber  Int
  blobUrl        String
  fileName       String
  fileSize       Int
  uploadedById   String
  createdAt      DateTime @default(now())

  document       Document @relation(fields: [documentId], references: [id])
  uploadedBy     User     @relation("VersionUploader", fields: [uploadedById], references: [id])
}
```

IMPORTANT: Use named relations "DocumentUploader" and "VersionUploader" on User to avoid Prisma ambiguous relation errors. Add corresponding backrelations on User model:
- `documentsUploaded Document[] @relation("DocumentUploader")`
- `versionsUploaded DocumentVersion[] @relation("VersionUploader")`

Also add NAMED backrelations on Customer, Order, and Batch models (Order already uses named relations like "CustomerOperatorOrders" — unnamed relations will cause Prisma ambiguous relation errors):
- Customer: `documents Document[] @relation("CustomerDocuments")`
- Order (the operator Order model): `documents Document[] @relation("OrderDocuments")`
- Batch: `documents Document[] @relation("BatchDocuments")`
And update Document model FK relations to match:
- `customer Customer? @relation("CustomerDocuments", fields: [customerId], references: [id])`
- `order Order? @relation("OrderDocuments", fields: [orderId], references: [id])`
- `batch Batch? @relation("BatchDocuments", fields: [batchId], references: [id])`

Run `npx prisma db push` after schema changes.

```typescript
import { z } from 'zod';

export const uploadDocumentSchema = z.object({
  name: z.string().min(1, { error: 'Document name is required' }).max(200),
  category: z.enum(['AGREEMENT', 'INVOICE', 'CERTIFICATION', 'SOP', 'MARKETING', 'OTHER'], {
    error: 'Invalid category',
  }),
  linkedTo: z.enum(['none', 'customer', 'order', 'batch']).optional().default('none'),

export type DocumentUploadFormState = {
  errors?: Record<string, string[]>;
  message?: string;
  success?: boolean;
};
```

Use `{ error: 'msg' }` NOT `{ message: 'msg' }` — this codebase is Zod v4.

Create `apps/command-center/src/app/actions/documents.ts` with 'use server' directive. Follow the established pattern from expenses.ts (verifySession, FormData handling, Vercel Blob put, revalidatePath).

**uploadDocument(prevState, formData)** — Server action for useActionState:
- Extract file from formData.get('file') as File. Validate file exists and size > 0.
- Extract and validate name, category, linkedTo, linkedId using uploadDocumentSchema.
- Guard: if no BLOB_READ_WRITE_TOKEN, log warning and return error (dev safety).
- Call `put('documents/${Date.now()}-${file.name}', file, { access: 'public', addRandomSuffix: true })`.
- Use `db.$transaction` to atomically create Document record and first DocumentVersion (versionNumber: 1).
- Set customerId/orderId/batchId based on linkedTo value (only one non-null).
- revalidatePath('/dashboard/documents').
- Return { success: true, message: 'Document uploaded successfully' }.

**uploadNewVersion(prevState, formData)** — Server action for uploading a new version:
- Extract documentId and file from formData.
- Query current max versionNumber for this document: `db.documentVersion.aggregate({ where: { documentId }, _max: { versionNumber: true } })`.
- Upload file to Vercel Blob with same pattern.
- Use `db.$transaction` to create new DocumentVersion (versionNumber = max + 1) and update Document.currentBlobUrl to new blob URL.
- revalidatePath('/dashboard/documents') and revalidatePath(`/dashboard/documents/${documentId}`).

**getDocuments(filters?)** — Fetch documents list:
- Accept optional filters: { category?, linkedTo?, search? }.
- Query db.document.findMany with where clause, include { uploadedBy: { select: { name: true } }, customer: { select: { businessName: true } }, order: { select: { orderNumber: true } }, batch: { select: { batchCode: true } }, _count: { select: { versions: true } } }.
- Filter out templates (isTemplate: false) unless explicitly requested.
- Order by updatedAt desc.

**getDocumentById(id)** — Fetch single document with all versions:
- Query db.document.findUnique with include { versions: { orderBy: { versionNumber: 'desc' }, include: { uploadedBy: { select: { name: true } } } }, uploadedBy: { select: { name: true } }, customer: true, order: true, batch: true }.

**deleteDocument(id)** — Soft delete or hard delete:
- Query document, verify it exists.
- Delete all DocumentVersion records for this document.
- Delete the Document record.
- Do NOT call del() on blob URLs (they're cached; research says avoid immediate blob deletion).
- revalidatePath('/dashboard/documents').

**getTemplates()** — Fetch template documents:
- Query db.document.findMany({ where: { isTemplate: true }, include: { versions: { orderBy: { versionNumber: 'desc' }, take: 1 } } }).

All actions must call verifySession() first. Import verifySession from '@/lib/dal', db from '@/lib/db', put from '@vercel/blob'.
