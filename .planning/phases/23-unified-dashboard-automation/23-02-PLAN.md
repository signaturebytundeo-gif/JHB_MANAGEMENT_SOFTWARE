**runSquareSyncInternal(userId: string | null): Promise<SyncResult>**

Copy the logic from `syncSquarePayments` in `marketplace-sync.ts` verbatim with the above three modifications. Import directly from the same integration libraries it uses (`getRecentSquarePayments` from `./square`, etc.).

**runEtsySyncInternal(userId: string | null): Promise<SyncResult>**

Copy the logic from `syncEtsyOrders` in `marketplace-sync.ts` verbatim with the above three modifications.

**runAmazonSyncInternal(userId: string | null): Promise<SyncResult>**

This function implements cursor-based micro-batching. It differs from `syncAmazonOrders`:

1. Check for an existing `RUNNING` MarketplaceSync record for AMAZON with a non-null `cursorData`. If found, this is a resumed batch run — use that sync record and extract `{ nextToken }` from `JSON.parse(syncRecord.cursorData)`.

2. If no existing RUNNING record with cursor, create a new one: `db.marketplaceSync.create({ data: { platform: 'AMAZON', status: 'RUNNING', triggeredById: userId } })`.

3. Get `sinceDate` using `getLastSuccessfulSyncDate('AMAZON')` — copy that helper from `marketplace-sync.ts` (it has no session check already).

4. Fetch ONLY ONE PAGE from the Amazon SP-API:
   ```ts
   const params: Record<string, string> = {
     MarketplaceIds: config.marketplaceId,
     CreatedAfter: sinceDate.toISOString(),
     OrderStatuses: 'Unshipped,PartiallyShipped,Shipped',
   };
   if (nextToken) params.NextToken = nextToken;
   const data = await spApiGet('/orders/v0/orders', params);
   ```
   Import `spApiGet` by re-exporting it from `amazon.ts` — or better: inline a local `spApiGet` using the same pattern (import `getAccessToken` and `getAmazonConfig` from `./amazon` if they are exported, otherwise duplicate the minimal fetch logic).

   Check exports in `amazon.ts`: if `spApiGet` and `getAccessToken` are not exported, implement a local version in `sync-internal.ts` that re-uses `getAmazonConfig()` (which IS exported from `config.ts`).

5. Process orders on this page using the same upsert + WebsiteOrder create logic from `syncAmazonOrders`.

6. Check `data.payload?.NextToken`:
   - If another page exists: update the sync record with `status: 'RUNNING'` and `cursorData: JSON.stringify({ nextToken: data.payload.NextToken })`. Return `{ success: true, message: 'Page processed, more pages remain', ... }`.
   - If no more pages: update sync record with `status: 'SUCCESS'` and `cursorData: null`, set `completedAt`. Return success result.

7. On error: set `status: 'FAILED'`, clear `cursorData`, return error result.

Export `SyncResult` type by re-exporting from `marketplace-sync.ts`: `export type { SyncResult } from '@/app/actions/marketplace-sync'`.

Import `getAmazonConfig`, `isPlatformConfigured` from `./config`. Import `getRecentSquarePayments` from `./square`, `getRecentEtsyOrders` from `./etsy`. Import `db` from `@/lib/db`.

**Step A — Create cron route handlers:**

Create three files. Each follows the exact same pattern:

```
apps/command-center/src/app/api/cron/sync-square/route.ts
```

**sync-square/route.ts:**
```typescript
import type { NextRequest } from 'next/server';
import { runSquareSyncInternal } from '@/lib/integrations/sync-internal';

export const maxDuration = 300;

export async function GET(request: NextRequest) {
  const authHeader = request.headers.get('authorization');
  if (authHeader !== `Bearer ${process.env.CRON_SECRET}`) {
    return new Response('Unauthorized', { status: 401 });
  }
  try {
    const result = await runSquareSyncInternal(null);
    return Response.json({ ok: true, result });
  } catch (error: any) {
    console.error('[cron/sync-square] Unhandled error:', error.message);
    return Response.json({ ok: false, error: error.message });
  }
}
```

**sync-amazon/route.ts:** Same structure but calls `runAmazonSyncInternal(null)`. Comment in file: `// Amazon uses cursor-based micro-batching — each invocation processes one page and exits.`

**sync-etsy/route.ts:** Same structure but calls `runEtsySyncInternal(null)`.

IMPORTANT: Return 200 (not 5xx) even on caught errors. Vercel does not retry failed cron invocations. Returning 5xx would just log a failure without retry benefit. The sync failure is already recorded in the MarketplaceSync table.

Do NOT add `export const dynamic = 'force-dynamic'` — GET handlers that read headers are already dynamic by default in Next.js App Router.
