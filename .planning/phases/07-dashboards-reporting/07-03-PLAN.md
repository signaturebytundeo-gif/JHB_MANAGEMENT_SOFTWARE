**reports.ts** — Create server actions for all 7 pre-built reports. Each function calls `verifySession()` first and accepts date range params.

1. **getDailySalesSummary(date: Date)** (RPT-01): Query `Sale` records for the given date grouped by `salesChannel.name`. Also query `Order` records (operator orders) for the same date grouped by channel. Return array of `{ channel, unitsSold, revenue, orderCount }`.

2. **getWeeklyProductionSummary(weekStart: Date)** (RPT-02): Query `Batch` records where createdAt is within the week (use `startOfWeek`/`endOfWeek` from date-fns). Group by product. Return `{ product, batchCount, totalUnits, releasedUnits, qcPassRate }`. QC pass rate = count of batches with status RELEASED / total batches * 100.

3. **getMonthlyPnL** — Delegate to existing `getPnLReport` from `financial-reports.ts`. Re-export or wrap it. (RPT-03 — already exists, just needs to be accessible from the reports page.)

4. **getInventoryValuationByLocation(date?: Date)** (RPT-04): Query `BatchAllocation` with product and location relations. For each (product, location) pair, calculate: available units (allocated - fulfilled from movements), unit cost (lowest PricingTier.unitPrice), total value. Group by location. Return `{ location, items: { product, units, unitCost, totalValue }[], locationTotal }[]`.

5. **getProductPerformance(startDate: Date, endDate: Date)** (RPT-05): Query `Sale` grouped by product, plus `OrderLineItem` grouped by product (with completed orders). For each product: total units sold, total revenue, average unit price, gross margin (revenue - estimated COGS using 40% wholesale proxy per Phase 3 convention). Return `{ product, unitsSold, revenue, avgPrice, estimatedMargin, marginPercent }[]`.

6. **getFarmersMarketPerformance(startDate: Date, endDate: Date)** (RPT-06): First query `SalesChannel` where `type IN ('EVENT', 'MARKETPLACE')` to get channel IDs (per research pitfall 6). Then query `Sale` + `Order` filtered by those channel IDs within date range. Group by channel name. Return `{ market, salesCount, revenue, avgOrderValue, topProduct }[]`.

7. **getSubscriptionMetrics()** (RPT-07): Query `SubscriptionMember` with plan relation. Calculate:
   - MRR: For each ACTIVE member, `billingCycle === 'annual' ? price / 12 : price` (per research pitfall 5)
   - Active count
   - Churn rate: CANCELLED / (ACTIVE + CANCELLED) * 100
   - LTV estimate: MRR * 12 (simple annual LTV)
   Return `{ mrr, activeCount, churnRate, ltv, memberDetails: { name, plan, status, monthlyValue }[] }`.

All functions return typed objects. Export a `ReportType` union type: `'daily-sales' | 'weekly-production' | 'monthly-pnl' | 'inventory-valuation' | 'product-performance' | 'farmers-market' | 'subscription-metrics'`.

**alerts.ts** — Create server actions for alert status:

1. **getAlertStatus()**: Check current state of 4 alert types:
   - **Low Inventory:** Query products where current stock (from BatchAllocation aggregation) < product.reorderPoint. Return list of `{ product, currentStock, reorderPoint, level: 'CRITICAL' | 'WARNING' }`.
   - **Overdue Invoices:** Query invoices where status is OVERDUE or (status SENT AND dueDate < now). Return `{ count, totalAmount, oldest }`.
   - **QC Failures:** Query batches from last 7 days with status HOLD. Return `{ count, batches: { code, product, date }[] }`.
   - **Pending Expense Approvals:** Query expenses where approvalStatus is PENDING. Return `{ count, totalAmount }`.

2. **checkAndSendAlerts()**: For each alert type with active issues, call Resend to send notification email to admin@jamaicahousebrand.com (using existing Resend pattern from customer-emails.ts). This is meant to be called as a side effect or manual trigger — NOT a cron job.

- Return the buffer as a response with correct Content-Type and Content-Disposition headers

**ReportSelector.tsx** — `'use client'` component. Renders a dropdown/select with 7 report options:
- Daily Sales Summary (default date: today)
- Weekly Production Summary (default: current week start)
- Monthly P&L (default: current month/year)
- Inventory Valuation by Location (no date needed)
- Product Performance (default: MTD range)
- Farmers Market Performance (default: MTD range)
- Subscription Metrics (no date needed)

Each option shows relevant date picker(s) when selected (single date for daily, week picker for weekly, month/year for P&L, date range for performance reports). Use shadcn/ui Select and Input components.

On selection change + date change, call `onReportChange(reportType, params)` callback. The parent page handles data fetching.

**ReportTable.tsx** — `'use client'` component. Generic data table that accepts:
- `columns: { key: string; header: string; format?: 'currency' | 'number' | 'percent' }[]`
- `data: Record<string, any>[]`
- `title: string`

Renders a responsive table (desktop) / card list (mobile) following the established pattern from `OperatorOrderList.tsx`. Currency values formatted with `$X,XXX.XX`, percentages with `X.X%`. Add a `print:block` wrapper class so the table prints cleanly. Include summary row at bottom for currency/number columns (totals).

**ExportButtons.tsx** — `'use client'` component. Props: `{ reportType: string; data: any[]; columns: Column[]; dateParams: Record<string, string> }`. Three buttons:
1. **CSV:** Client-side Blob construction (copy pattern from CSVExportButton.tsx). Build CSV string from columns + data, create Blob, trigger download. Filename: `{reportType}-{date}.csv`.
2. **PDF:** `window.print()` button. Add `print:hidden` to the buttons row itself. The ReportTable visible content prints as the PDF.
3. **Excel:** `fetch('/api/export/excel?report=${reportType}&${dateParams}')` then create a Blob from the response and trigger download. Filename from Content-Disposition header.

All three buttons in a row with icons (FileText for CSV, Printer for PDF, FileSpreadsheet for Excel from lucide-react).

**AlertConfigPanel.tsx** — Server component. Receives alert status data as props from the page. Renders 4 cards:
1. **Low Inventory:** Shows count of products below reorder point. Lists product names with current vs threshold. Badge: CRITICAL (red) if any at 0, WARNING (amber) otherwise, OK (green) if none.
2. **Overdue Invoices:** Shows count and total overdue amount. Badge by count.
3. **QC Failures (7 days):** Shows count of batches on HOLD. Lists batch codes.
4. **Pending Approvals:** Shows count and total amount of pending expenses.

Each card shows current thresholds as read-only text (e.g., "Reorder threshold: per-product setting", "Overdue: past due date", "QC: auto-hold on fail"). These are hardcoded defaults per research recommendation — no editable threshold UI for Phase 7.
