Add three new enums to the Prisma schema:

1. `OperatorOrderStatus` with values: DRAFT, CONFIRMED, PROCESSING, SHIPPED, DELIVERED, COMPLETED, CANCELLED
   - Do NOT modify the existing `OrderStatus` enum (used by WebsiteOrder). Create this as a completely separate enum.

2. `InvoiceStatus` with values: DRAFT, SENT, VIEWED, PARTIAL, PAID, OVERDUE, VOID

3. `OperatorOrderType` with values: STANDARD, CATERING, FARMERS_MARKET

Add four new models exactly as specified in 04-RESEARCH.md:

1. `Order` model with fields:
   - id (cuid), orderNumber (unique String), status (OperatorOrderStatus default DRAFT), orderType (OperatorOrderType default STANDARD)
   - customerId (optional), channelId, locationId, paymentMethod (PaymentMethod enum)
   - orderDate (DateTime default now()), notes (optional String)
   - Catering fields: depositAmount (Decimal?), depositPaidAt (DateTime?), balanceDueDate (DateTime?), eventDate (DateTime?)
   - Farmers market fields: eventLocation (String?), weatherNotes (String?), footTrafficNotes (String?)
   - Approval fields: totalAmount (Decimal default 0), approvalStatus (String?), approvedById (String?), secondApprovedById (String?), approvedAt (DateTime?)
   - Audit: createdById, createdAt, updatedAt
   - Relations: customer (Customer?), channel (SalesChannel), location (Location), createdBy (User "OrderCreator"), approvedBy (User? "OrderApprover"), secondApprovedBy (User? "OrderSecondApprover"), lineItems (OrderLineItem[]), invoice (Invoice?), inventoryMovements (InventoryMovement[] "OrderMovements")
   - Indexes: status, customerId, channelId, orderDate, orderType

2. `OrderLineItem` model with: id, orderId, productId, quantity (Int), unitPrice (Decimal), totalPrice (Decimal). Relations to Order (onDelete Cascade) and Product. Indexes on orderId, productId.

3. `Invoice` model with: id, invoiceNumber (unique), orderId (unique), customerId (optional), status (InvoiceStatus default DRAFT), subtotal, taxAmount (default 0), totalAmount, paidAmount (default 0), issuedAt, dueDate, paidAt, viewedAt, overdueAt, lateFeeAmount (default 0), notes, createdAt, updatedAt. Relations to Order, Customer?, InvoicePayment[]. Indexes on status, customerId, dueDate, issuedAt.

4. `InvoicePayment` model with: id, invoiceId, amount (Decimal), paymentMethod (PaymentMethod), paidAt (DateTime), notes, createdById, createdAt. Relations to Invoice and User. Indexes on invoiceId, paidAt.

Add to the existing `InventoryMovement` model:
- `orderId String?` field
- `order Order? @relation("OrderMovements", fields: [orderId], references: [id])` relation

Add reverse relations on existing models:
- `Customer`: add `orders Order[]` and `invoices Invoice[]`
- `SalesChannel`: add `orders Order[]`
- `Location`: add `orders Order[]` (use named relation if Location already has other Order-like relations)
- `Product`: add `orderLineItems OrderLineItem[]`
- `User`: add `createdOrders Order[] @relation("OrderCreator")`, `approvedOrders Order[] @relation("OrderApprover")`, `secondApprovedOrders Order[] @relation("OrderSecondApprover")`, `invoicePayments InvoicePayment[]`

After schema changes, run `npx prisma db push` (not migrate, matching project pattern) to apply. Then run `npx prisma generate` to regenerate the client.

Create `apps/command-center/src/lib/validators/operator-orders.ts`:
- `createOperatorOrderSchema`: z.object with channelId (string min 1), locationId (string min 1), customerId (optional string), paymentMethod (z.enum of PaymentMethod values), orderType (z.enum STANDARD/CATERING/FARMERS_MARKET default STANDARD), notes (optional string), lineItems (z.string().transform to parse JSON array of {productId: string, quantity: number, unitPrice: number}). Add z.refine to ensure lineItems array has at least 1 item after parsing.
- Catering fields (all optional): depositAmount, eventDate, balanceDueDate
- Farmers market fields (all optional): eventLocation, weatherNotes, footTrafficNotes
- Export `CreateOperatorOrderFormState` type: `{ errors?: Record<string, string[]>; message?: string; success?: boolean; orderId?: string } | undefined`
- `updateOrderStatusSchema`: z.object with orderId (string), status (z.enum of OperatorOrderStatus values)

Follow the exact pattern from `apps/command-center/src/lib/validators/sales.ts` for FormState type and Zod v4 syntax (use 'error' key not 'required_error').

Create `apps/command-center/src/lib/validators/invoices.ts`:
- `createInvoiceSchema`: z.object with orderId (string min 1), notes (optional string), taxRate (z.coerce.number().min(0).max(100).default(0))
- `logPaymentSchema`: z.object with invoiceId (string min 1), amount (z.coerce.number().positive()), paymentMethod (z.enum of PaymentMethod values), paidAt (z.coerce.date()), notes (optional string)
- Export `CreateInvoiceFormState` and `LogPaymentFormState` types following same pattern.

Create `apps/command-center/src/lib/utils/order-number.ts`:
- `generateOrderNumber(tx?)`: Accepts optional Prisma client/transaction. Queries count of orders created in current year (createdAt >= Jan 1 of current year). Returns `ORD-${year}-${String(count + 1).padStart(4, '0')}`.
- `generateInvoiceNumber(tx?)`: Same pattern but for Invoice model. Returns `INV-${year}-${String(count + 1).padStart(4, '0')}`.
- Import db from `@/lib/db` for default client.
